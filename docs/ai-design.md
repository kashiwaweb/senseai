# senseai AI設計書

最終更新: 2026-05-01

> **このドキュメントは senseai プロジェクトの心臓部です。**
> AIの出力品質がそのまま製品価値になるため、PoC期間中は継続的に改善します。

## 1. 設計方針
1. **前向き**: フィードバックは必ず「ダメ出し」ではなく「次にこうしてみよう」
2. **具体的**: 抽象的な助言ではなく、必ず具体的なフレーズや行動例を提示
3. **構造化出力**: AI出力は厳密な JSON スキーマで取得し、UIで解釈
4. **エビデンスベース**: 「魔法の言葉」など具体例は必ず実データから引用（捏造禁止）
5. **モデル切替可能**: プロバイダー抽象化レイヤを設け、品質とコストで切替

## 2. モデル選定
| ユースケース | 第一候補 | フォールバック | コスト目安 |
|---|---|---|---|
| 文字起こし | Groq Whisper API（Large-v3-Turbo） | テキスト直接アップロード / whisper.cpp ローカル / OpenAI Whisper API | 無料枠 |
| 「型」抽出（フェーズ①） | Gemini 2.5 Flash | Claude Haiku 4.5 | 無料枠 |
| 診断生成（フェーズ②） | Gemini 2.5 Flash | Claude Sonnet 4.6 | 無料枠 |

**文字起こし**: 音声/テキストの2モードを最初から両対応（詳細は `architecture.md §2.2`）。Groq の無料枠を超えた場合や Groq 障害時はテキストモードを案内。

**切替判断基準**: PoC初期はGeminiで開始。オーナーレビューで以下のいずれかが起きたらClaude切替:
- 出力フォーマット崩れ（スキーマ違反）が頻発
- フィードバック文のトーンが「冷たい」「ダメ出しに聞こえる」
- 日本語のニュアンスがエステ業界に合わない

## 3. 7ステップの定義（共通）
| # | ステップ | 観察ポイント |
|---|---|---|
| 1 | 安心を作る | 第一声、笑顔（言外の温度感）、名乗り、来店ねぎらい、コートの預かりなど |
| 2 | 想いを引き出す | 質問の数・深さ、潜在ニーズへの掘り下げ、悩みの確認 |
| 3 | 心に寄り添う | 共感の言葉、お客様の感情の反復、「わかります」の使い方 |
| 4 | プロとしての提案 | 根拠ある提案、選択肢の提示、お客様の状態に基づく判断 |
| 5 | 施術中コミュニケーション | 状況確認（強さ・温度）、リラックスを促す会話、間合いの取り方 |
| 6 | 納得のクロージング | 施術内容のサマリ、効果の説明、安心感、次回までのケア |
| 7 | 未来への期待 | 次回提案、ホームケア商品、お見送り、リピート導線 |

### 3.1 横断的な観察レンズ（プロンプトに組み込む）
7ステップの評価時、AIは以下の「能力レンズ」を意識して各ステップの強み・伸びしろを言語化する。
独立スコアとしては保持しないが、`description` / `strengths` / `growth_areas` などのコメント文に自然な日本語で織り込む。

| 観察レンズ | 観察内容 |
|---|---|
| 共感力 | 感情ワードを拾えているか、「わかります」「気になりますよね」の使い方 |
| 深掘り力 | 質問の深さ、潜在ニーズへの掘り下げ、WHY を重ねる質問 |
| 提案力 | 根拠ある提案、選択肢の自然な提示、お客様の状態に基づく判断 |
| 会話テンポ | 沈黙・割り込みのバランス、間の取り方 |
| 成約導線 | クロージングの自然な流れ、押し付けにならない次回提案 |

→ 縦軸（7ステップ）と横軸（観察レンズ）のマトリクス評価は PoC では行わない。観察レンズはあくまで AI のコメント生成を厚くするための「視点」として使う。

## 4. フェーズ① プロンプト戦略

### 4.1 入力データ
種別タグ付きセッション群:
- `customer`: 顧客接客の会話書き起こし
- `instruction`: スタッフへの指導の音声書き起こし
- `philosophy`: 心掛けスピーチの書き起こし

3種類を組み合わせることで「行動（customer）」「教え（instruction）」「価値観（philosophy）」を立体的に把握する。

### 4.2 システムプロンプト要点
- 役割: エステサロンの接客育成コンサルタント
- 目的: 提供された資料からトップスタッフの「型」を構造化して言語化
- 7ステップの定義を提示
- `magic_words` は実際にトップが使った具体的フレーズを抽出（捏造禁止）
- 出力は JSON Schema に厳密に従う（コードブロック・前置き禁止）

### 4.3 出力 JSON スキーマ
```json
{
  "overview": "スーパースタッフの全体的な特徴（200字程度）",
  "core_values": ["価値観を表す短文", "..."],
  "tone": {
    "speaking_style": "丁寧語の使い方など",
    "pace": "話すスピード・間",
    "emotional_register": "声色・温度感"
  },
  "seven_steps": [
    {
      "step_number": 1,
      "step_name": "安心を作る",
      "description": "このステップでスーパースタッフが大切にしていること（150字）",
      "key_behaviors": ["行動1", "行動2", "..."],
      "magic_words": [
        {
          "phrase": "実際に使われたフレーズ",
          "intent": "なぜこの言葉が効くのか",
          "context": "どんな場面で使うか",
          "source_session_id": "引用元セッションID"
        }
      ],
      "common_pitfalls": ["やりがちなミス（自店スタッフ向け注意点）"]
    }
  ]
}
```

### 4.4 プロンプト草稿（v0）
```
あなたはエステサロンの接客育成コンサルタントです。
以下の3種類の音声書き起こしを分析し、このスーパースタッフの「型」を構造化してください。

# 入力データ
{sessions_serialized}

# 7ステップ定義
1. 安心を作る - 第一声、ねぎらい、空気作り
2. 想いを引き出す - ヒアリング、潜在ニーズの掘り下げ
3. 心に寄り添う - 共感
4. プロとしての提案
5. 施術中コミュニケーション
6. 納得のクロージング
7. 未来への期待 - 次回提案、お見送り

# 出力ルール
- 必ず以下のJSONスキーマに従うこと
- magic_words は実際の発話から原文ママで引用すること（捏造禁止）
- 引用がない場合は magic_words を空配列にすること
- 出力は JSON のみ。前置き・コードブロック・説明文は禁止

# JSONスキーマ
{json_schema}
```

## 5. フェーズ② プロンプト戦略

### 5.1 入力データ
- 最新の TopProfile JSON
- 一般スタッフの会話書き起こし（1セッション）
- スタッフ名・経験年数（任意）

### 5.2 システムプロンプト要点
- 役割: 若手スタッフを優しく導く先輩トレーナー
- 「型」と会話を照らし合わせ、7ステップごとに評価
- スコアは 1-5。「3」が標準。「1-2」も「未習得」ではなく「これから伸ばすステップ」と表現
- `improvement_examples` の `top_would_say` は型の `magic_words` から関連フレーズを引用、なければ生成
- 結びは必ず励ましのメッセージ
- 否定語（「できていない」「ダメ」「足りない」）は使わない

### 5.3 出力 JSON スキーマ
```json
{
  "overall_summary": "全体評価（前向きな150字）",
  "encouragement_message": "個別の励ましメッセージ（80字）",
  "average_score": 3.4,
  "seven_steps": [
    {
      "step_number": 1,
      "step_name": "安心を作る",
      "score": 4,
      "score_label": "得意",
      "strengths": [
        {
          "observation": "良かった点の説明",
          "quote": "実際の発話の原文引用"
        }
      ],
      "growth_areas": [
        {
          "observation": "伸ばせる点の説明（前向きな表現）",
          "suggestion": "次にやってみるとよいこと"
        }
      ],
      "improvement_examples": [
        {
          "your_words": "あなたが言った言葉",
          "top_would_say": "トップならこう言う",
          "why": "理由"
        }
      ]
    }
  ]
}
```

### 5.4 スコア基準
| スコア | ラベル | 意味 |
|---|---|---|
| 5 | お手本 | トップに近いレベルで実践できている |
| 4 | 得意 | 強みとして伸ばしていきたい |
| 3 | 標準 | できている、改善でさらに良くなる |
| 2 | これから伸ばす | 次の練習テーマにしよう |
| 1 | 次の一歩 | 取り組むと大きく成長できるポイント |

## 6. 失敗時のフォールバック
| 失敗ケース | 対応 |
|---|---|
| LLM出力がスキーマ違反 | Zodで検出 → 1度自動リトライ → ダメなら raw を保存し UI に「再生成」ボタン |
| LLM API タイムアウト（30秒） | UI上は「分析中...」維持・バックグラウンド再試行 |
| 文字起こしが空・極端に短い | 診断スキップ・ユーザーに音声品質確認を促す |
| `magic_words` が0件 | 警告だけ表示し続行（録音セッション数を増やすよう促す） |

## 7. プロンプトバージョン管理
- プロンプトはコード内でバージョン文字列を持つ（例: `prompt_phase1_v1`）
- TopProfile / Diagnosis レコードに `prompt_version` を保存
- 比較しやすいよう、過去バージョンも `server/lib/prompts/` に残す

## 8. プロンプト改善ループ（PoC で必須）
1. プロンプト v1 → 実データで実行
2. オーナーに結果を見てもらいフィードバック取得
3. 「ここをこう言ってほしい」を反映してプロンプト改善
4. v2, v3 と繰り返す
5. 採用版が固まったらドキュメントに転記

## 9. プライバシー処理
- 文字起こし後、固有名詞（顧客名・電話番号・住所）を正規表現＋LLMで `[氏名]` などにマスク
- LLMに送るのはマスク後のテキスト
- マスク機能は W2 以降に実装（W1ではダミーデータで進める）

## 10. オープンクエスチョン
以下は今後の議論で決めたい点:
- ✏️ スコア基準の解釈をオーナーと擦り合わせる必要あり
- ✏️ 「型」を生成するための最低セッション数（3? 5? 10?）
- ✏️ 同じスタッフを複数回診断する場合、過去診断との成長比較を出すか
- ✏️ お客様の発話と スタッフの発話 をどう区別するか（話者分離が必要か、推定で十分か）
- ✏️ 1セッションあたりの最大文字数・LLMコンテキスト長の運用ルール

## 11. 設計判断の記録（PoCスコープ）
PoC期間中は以下を採用しない。本番フェーズで価値検証後に再考する。

| 却下した選択肢 | 理由 | 再考の条件 |
|---|---|---|
| 5評価軸を独立スコアとして持つ | 縦軸=7ステップが既にオーナー説明済み。マトリクス（7×5=35セル）はオーナー視点で複雑。観察レンズとしてプロンプトに組み込めば代替可能（§3.1） | レーダーチャートで「能力別」可視化が強く求められた場合 |
| 構造化データ層（3層構造の中央層） | 発話単位ラベリングは AIタスクとして大きく、PoC期間で精度検証が完了しない。生データ＋抽象化の2層で十分 | 「発話頻度の時間推移」など細かい統計需要が出たとき |
| 発話単位スコアリング（intent/emotion/score） | 1セリフ単位は AI判定がブレやすく、価値検証前に開発コストが膨らむ | ヒートマップ可視化や瞬間的な指摘需要が出たとき |
| 自動セグメント分類（カウンセリング/クロージング/雑談を別テーブルで管理） | 7ステップが既に時間的セグメント概念を内包。AIに7ステップ評価を任せれば内部で該当箇所を判断 | 「クレーム対応の場面だけ抽出」など横断機能が必要になったとき |
